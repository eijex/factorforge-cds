# 2026-07-05 — 연구노트(Research Note) 포맷 도입 검토

**계기**: 반복되는 작업 중 상당수가 `_improvement-jobs`(status 기반 작업 대장)에 올리기엔 애매하다고 느낌. 완료된 기능이 아니라 "왜 그렇게 판단했는지"의 과정 자체가 남는 자리가 없었음.

**상황**: `_improvement-jobs/`는 "무엇을 만들었나", `_analysis/`는 "무엇을 알아냈나"를 기록하는데, 그 사이에 있는 서술형 판단 기록(고민→결론)을 남길 세 번째 자리가 없었음. 여기에 더해, 이런 기록 일부를 eijex.com에도 공개하고 싶다는 목표가 있었음.

**고민한 것**:
- 형식: 일기(diary) / changelog / 랩노트(lab notebook) 중 어느 쪽이 맞는가
- 참고할 기존 컨벤션: ADR(Nygard/MADR 포맷, "왜 이렇게 결정했나"에 특화), log4brains(ADR을 퍼블리시 가능한 사이트로 자동 빌드), TIL repo 패턴(결정보다 가벼운 일상 기록)
- 저장 위치: 저장소 `docs/` 안에 둘 것인가, eijex.com에 바로 올릴 것인가
- 공개 리스크: 외부 협력사로부터 받은 데이터, 미래 제품명이 폴더명·파일명 수준에서도 노출될 수 있음

**결론**:
- 포맷은 ADR에 가까운 **계기 → 상황 → 고민한 것 → 결론 → 남은 질문/다음** 구조로 시작. 완벽한 템플릿을 미리 만들지 않고 몇 개 써본 뒤 다듬기로 함.
- 저장은 2단계로 분리:
  1. **1차(내부)**: 각 저장소 `docs/research-notes/`에 편하게 기록 — 기존 MkDocs + GitHub Pages 파이프라인을 그대로 재사용 (`docs/roadmap.md`, `docs/changelog.md`와 동일한 패턴)
  2. **2차(공개, 나중)**: 쌓인 노트 중 선별한 것만 eijex.com "Research Notes" 섹션으로 옮겨 공개. 이 "옮기는 행위" 자체가 공개 전 검수 게이트 역할을 하도록 설계.
- 공개 채널(GitHub 포함)에는 외부 협력사 소유 데이터·미래 제품명·특정 협력사/프로젝트를 식별 가능하게 하는 폴더명·파일명을 절대 포함하지 않음.

**남은 질문 / 다음**:
- mkdocs-material의 blog 플러그인이 현재 설치 버전에서 지원되는지 확인 필요 (지원 시 날짜순 아카이브 자동 생성)
- `mkdocs.yml` nav에 정식으로 추가할지는 몇 개 더 써본 뒤 결정
- eijex.com Research Notes 섹션 설계는 별도 단계로 보류
- 이 트라이얼이 유효하다고 판단되면 내부 작업 관리 체계에 정식 등록 여부 결정
